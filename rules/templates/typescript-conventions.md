## TypeScript Config

- **Strict mode**: enabled (`"strict": true` in tsconfig)
- No implicit `any` ‚Äî all values must be typed
- Prefer `interface` for object shapes, `type` for unions/intersections/aliases
- Avoid `as` type assertions ‚Äî use type guards or proper typing instead
- Use `unknown` instead of `any` at system boundaries (external APIs, JSON parsing)

## State Management (Angular only)

- **Default to `signalStore`** (`@ngrx/signals`) for all new state ‚Äî no new `BehaviorSubject` stores
- `@ngrx/signals` major version must match Angular major (Angular 20 -> `@20`, Angular 21 -> `@21`)
- Use `withState`, `withComputed`, `withMethods`, `patchState` ‚Äî never mutate state directly
- Derived state goes in `withComputed`, not component logic
- Read signals directly in templates (`{{ store.prop() }}`), not via `async` pipe
- Use `toSignal` / `toObservable` from `@angular/core/rxjs-interop` at HTTP/RxJS boundaries
- **Touching an existing store for any reason? Migrate it to `signalStore`** in the same piece of work ‚Äî use a bridge service to preserve the API surface without touching every consumer at once
- Full patterns, migration map, and before/after examples: `rules/templates/ngrx-signals.md`

## Testing

- **Framework**: [DETECTED: vitest | jest | mocha]
- Test files: co-located `*.test.ts` or in `__tests__/` directory
- Naming: `describe('<unit>')` + `it('should <behaviour>')`
- Structure: Arrange / Act / Assert
- Mock external calls ‚Äî do not make live HTTP requests in unit tests
- Use `vi.mock()` (vitest) or `jest.mock()` for module mocks
- Integration tests: use `supertest` for HTTP layer testing (if detected)

## Async

- Use `async`/`await` over raw Promises
- Handle errors with `try/catch` ‚Äî do not leave floating `.catch()` handlers
- Use `Promise.all()` for concurrent independent operations

## Imports

- Absolute imports over deep relative paths ‚Äî configure `paths` in tsconfig
- Group imports: external packages -> internal modules -> types
- No circular imports

## Anti-Patterns

- No `@ts-ignore` or `@ts-expect-error` without an explanatory comment
- No `console.log` in production code ‚Äî use a structured logger
- No non-null assertion (`!`) on values that could legitimately be null
- No `any` ‚Äî use `unknown` and narrow, or model the type properly
