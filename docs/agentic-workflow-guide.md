# Agentic Claude Code Workflow Guide

## Overview

A document-driven workflow for using Claude Code as an AI pair programmer. Built on human-in-the-loop principles with architectural governance and iterative verification. Every significant change is preceded by a decision record and followed by a committed execution plan.

---

## Core Principles

| Principle | Description |
|-----------|-------------|
| **Human-in-the-Loop** | Claude proposes; the architect approves. Collaboration, not autonomous execution. |
| **Plan-and-Solve** | Claude generates an execution plan before any implementation code. |
| **Self-Refine** | Claude critiques its own output against the spec before human review. |
| **Context Isolation** | Load only relevant files. Never load the entire docs folder. |
| **Traceable Decisions** | All architectural choices captured in versioned ADRs. |

---

## Trust Ladder Framework

| Level | Name | Claude Capabilities | Human Review Required |
|-------|------|---------------------|----------------------|
| 1 | Read-Only | Analyse code, explain logic | Mental review only |
| 2 | Propose & Review | Read files, propose changes | Review all changes before commit |
| 3 | Implement & Verify | Write code, run tests, create commits | Review commits, approve PRs |
| 4 | Guided Autonomy | Plan, implement, test within approved bounds | Periodic review, approve milestones |

*Recommended starting level: 2-3 (Propose & Review / Implement & Verify)*

## Document-Driven Workflow (ADR-Based)

### File Structure

```
project-root/
├── CLAUDE.md                          # Workflow entry point (auto-loaded)
├── .claude/
│   ├── adr-kit/                       # Submodule: shared workflow, commands, rules
│   ├── commands/                      # Command shims (auto-generated by adr-init)
│   ├── rules/                         # Rules (copied from adr-kit by adr-init)
│   └── settings.json                  # Repo-specific permissions
└── docs/
    ├── adr-index.md                   # Topic-to-ADR lookup (no full ADR reading)
    ├── architecture.md                # C4 model + decision log table
    ├── adrs/
    │   └── NNN-title-adr.md           # Architecture Decision Records
    ├── execution-plans/
    │   ├── NNN-execution-plan.md      # Active plan (living document)
    │   └── archive/                   # Completed plans
    └── backlog/                       # Deferred work candidates
```

## Workflow Steps

1. **Requirement Handoff**: Describe the feature or change -> Claude generates clarifying questions in the ADR.
2. **Clarification Phase**: Answer questions directly in the ADR markdown.
3. **Tradeoff Analysis**: Claude proposes options; architect selects in ADR.
4. **ADR Acceptance**: Set Status: Accepted.
5. **Execution Plan**: Claude creates a detailed execution plan as an approval gate.
6. **Plan Approval**: Review and approve the execution plan.
7. **Implementation**: Claude generates code only after execution plan approval.
8. **Verification**: Review code and run tests; Claude iterates if needed.
9. **Documentation**: Claude updates ADR with results, archives execution plan.
10. **Architecture Update**: Claude adds row to decision log in architecture.md.

---

## Document Polish Convention

**ADR** = Final decision document. Clean, formal, polished. No back-and-forth visible.

**Execution Plan** = Has discussion sections where decisions are made. Claude **replaces** raw inline architect comments with clean summaries that preserve the decision.

**Rule**: When the architect provides inline feedback in any document:
1. Read and understand — capture the decision, rationale, and constraints
2. Replace the raw inline comments with a clean summary
3. Preserve the decision exactly — never change what was decided, only how it's expressed
4. Improve clarity — fix grammar, expand abbreviations, add context

## Session State Management

### Two-Phase Session State Model

| Phase | State Location | Purpose |
|-------|----------------|---------|
| **ADR Development** (Status: Proposed) | ADR "Session State" section | Track clarification/approval progress |
| **Implementation** (Status: Accepted) | Execution Plan | Track coding progress |

### CLAUDE.md Active Work Pointer

**Purpose**: Single entry point for new agents to find current work.

**Agent Resume Flow**:
```
New Session Starts
    |
Agent loads CLAUDE.md (automatic)
    |
See "Active Work" section
    |
Load referenced ADR
    |
Check ADR Status field:
    |- Status: Proposed -> Read ADR "Session State" section
    +- Status: Accepted -> Load Execution Plan -> Read "AGENT: START HERE"
```

## Execution Plan as Session State

### The Problem: Stateless Agents

Agents have no memory between sessions. The execution plan is a **persisted session state** stored in git.

### Agent Initialization Flow

```
Session Start
    |
Load CLAUDE.md (automatic)
    |
Read "Active Work" section
    |
Load ADR -> Check Status
    |
    |- Status: Proposed
    |   Read "Session State" in ADR
    |   Know: Phase, Waiting On, Next Action
    |   Update ADR + commit
    |
    +- Status: Accepted
        Load Execution Plan
        Read "AGENT: START HERE" (first 50 lines)
        Know: Status, Current Step, Blocker, What to Load
        Load ONLY files listed in "What Do I Load?"
        Execute "What's My Next Action?"
        Update execution plan + commit
```

### Tight Context Optimization

When context is tight:
1. Load execution plan (~500 lines)
2. Read "Agent Quick Start" (~50 lines)
3. Load ONLY files in "What Do I Load?" (2-4 files)
4. **Total context**: ~1500 lines vs 10,000+ from exploring the repo

This is roughly 7x more efficient than loading the entire docs/ folder.

## Parallel Agent Coordination

### When to Use Parallel Work

Multiple agents can work simultaneously on different ADRs when:
- Agent A implements a service, Agent B writes tests for it
- Agent A builds an API layer, Agent B builds the client
- Agent A refactors core, Agent B adds a new feature using existing interfaces

### Parallel Work Brief Document

**When to create**: When parallel work is approved on two ADRs that touch related code.

**Location**: `docs/parallel-work-brief-ADR-X-ADR-Y.md`

**Template**: `@.claude/adr-kit/docs/templates/parallel-work-brief-template.md`

**Key sections**:
1. Who's working on what and why parallel
2. Agent A constraints (what not to break for Agent B)
3. Agent B constraints (what not to break for Agent A)
4. File ownership boundaries
5. Frozen interface contracts
6. Merge strategy (pull/push discipline)

### Coordination Rules

- Pull at session start — always
- Push at session end — always
- Push immediately after interface changes — don't leave breaking changes unpushed
- One agent owns one directory tree — avoid overlapping file edits

## Backlog Workflow

`docs/backlog/` contains candidate files for deferred work.

**Lifecycle**: Candidate -> Promoted to ADR -> Done (delete file) | Dropped (delete + commit reason)

**Cleanup**: When done, delete the file. The commit message and/or ADR are the permanent record.

## Prompt Patterns

### Session Start Template

```
Load @.claude/adr-kit/docs/agentic-workflow-guide.md
I want to [TASK].
Follow ADR workflow. Don't implement until execution plan is approved.
```

### Clarification Prompt

```
Review the ADR Context section. Generate 5-10 clarifying questions
focused on: security, data volume, error handling, deployment, and validation.
Format output as a Markdown table for direct insertion into the ADR.
```

### Implementation Prompt (Post-ADR Acceptance)

```
Implement [component] per ADR-NNN.
Output:
1. Unified diff format for existing files
2. Full file content for new files (with path header)
3. Proposed unit tests
4. Verification steps (run tests, lint, typecheck)
```
